Byte vs Block Addressable Memory & B-Trees

An interactive guide to how RAM, Disk, and Databases work together

The Big Picture

When your faculty said "RAM is byte addressable and Disk is block addressable", they were pointing at a very deep idea that shapes how databases are designed.

RAM (Memory)

  • Lives on a stick attached to the CPU
  • Access time: ~100 nanoseconds
  • Can fetch any single byte directly
  • Volatile - loses data on power off
  • Expensive per GB

Disk (HDD / SSD)

  • Persistent storage
  • Access time: ~10 milliseconds (HDD), ~100µs (SSD)
  • Cannot fetch one byte - reads entire blocks/pages (usually 4 KB or 8 KB)
  • Survives power off
  • Cheap per GB
Speed gap intuition: If a RAM read takes 1 second, a disk read takes about 1 day. That's 100,000× slower. Every database optimization basically tries to not touch the disk when possible, and when it must, to grab as much useful data as possible in one shot.

Why does this matter for arrays?

You said: arr[i] is O(1). That's true only when the array is in RAM. The CPU computes baseAddress + i * elementSize and fetches that byte directly. Easy.

But if your "array" is a 1 TB table sitting on disk, you cannot just compute an address and fetch one byte. The disk physically refuses to give you less than a whole block. That single restriction is why databases use B-Trees instead of arrays or hashmaps.

Byte Addressable Memory (RAM)

Every byte in RAM has its own unique numeric address. The CPU can ask the memory controller "give me byte at address X" and get it back in roughly 100 nanoseconds.

How arr[i] works in O(1)

// Suppose arr starts at address 1000, each int = 4 bytes
arr[0] -> address 1000
arr[1] -> address 1004
arr[2] -> address 1008
arr[i] -> address 1000 + i * 4   <-- just arithmetic, one fetch

Block Addressable Memory (Disk)

Disks were designed for bulk storage, not random access. Physical reasons (spinning platters on HDDs, flash page sizes on SSDs) force the disk to operate in chunks called blocks (also called pages in DB literature).

A typical block is 4 KB or 8 KB. Even if you only want 1 byte, the disk reads the whole 4096 bytes containing it.

The painful reality

// You want byte at offset 5000 on disk
// Disk: "I can only give you blocks. Byte 5000 lives in block 1 (bytes 4096..8191)"
// You get back 4096 bytes. The other 4095 are 'wasted' work for this query.
// AND the whole operation took ~10ms (100,000x slower than RAM)

The DBMS Problem

Imagine a users table with 1 billion rows. Total size: 200 GB. RAM size: 16 GB. You cannot fit it all in memory. The table must live on disk.

Query: SELECT * FROM users WHERE id = 7382910;

Naive approach #1: Linear scan

Read every block, check every row. With 50 million blocks × 10ms = ~6 days per query. Unacceptable.

Naive approach #2: Sorted array on disk + binary search

Better - O(log n) reads, ~30 disk hits = ~300 ms. But each hit potentially jumps to a different block. And inserts are painful (shifting billions of rows).

💡 Wait - why do blocks "jump around"? Aren't they stored continuously on disk?

Great question. The short answer: data on disk is almost never stored in one continuous stretch. Let's build the intuition step by step.

📚 The library analogy (the easiest way to get this)

Imagine a giant library with thousands of shelves. Each shelf holds exactly 10 books. The shelves are your disk blocks. The books are your rows.

On Day 1, the library is empty. You receive 30 books for "Topic A" and place them neatly on shelves #1, #2, #3. Continuous. Perfect.

On Day 2, you get 20 books for "Topic B" and put them on shelves #4, #5.

On Day 3, your boss gives you 15 more books for Topic A. Shelves #1, #2, #3 are full. Shelf #4 belongs to Topic B. Where do you put the new Topic A books?

→ You put them on shelf #6 and shelf #7. Now Topic A's books live on shelves 1, 2, 3, 6, 7 - no longer continuous.

On Day 4, someone removes 5 books from shelf #2. Now shelf #2 has gaps. Later you get new books for Topic C - they fill those gaps on shelf #2. So shelf #2 is now mixed.

This is exactly what a hard disk looks like after a few months of real use - a patchwork of blocks belonging to many different files, scattered everywhere.

❓ But how does the database know "ID 9-16 is at physical block #7"?

The file system keeps a translation table (on Linux, this is the inode; on Windows, the MFT). When the DB asks "give me byte 8000 of my data file", the OS looks up: "byte 8000 of users.dat lives at physical block #7 of the drive", and fetches that one block.

This is invisible to the DB - but the cost is very real.

⏱️ Why "jumping blocks" is so expensive (HDD specifically)

On an HDD, the head physically moves to find the right track:

  • If you read block #2, then block #3, then block #4 (continuous) - head barely moves. Each read ~0.1 ms after the first.
  • If you read block #2, then block #7, then block #10, then block #5 (scattered) - head must seek each time. Each seek ~5-10 ms.

So binary search on a sorted "array" stored on disk doesn't get you the speed of an in-RAM binary search. Each comparison forces a jump to a random block - every comparison is a 10 ms seek.

🤔 Hold on - what exactly is "the head"? If the system already knows the address, why does location matter?

This is the single most important physical fact to internalize. Your mental model is right for RAM but wrong for HDDs. Let's fix it.

📷 What a hard disk actually looks like inside

Pop open an HDD and you'd see something that looks like a tiny record player:

HDD anatomy (animated)

The pink square is the read/write head - a microscopic sensor at the end of the arm that detects/changes the magnetic polarity of tiny spots on the platter.

The blue circles are tracks - concentric rings where data lives.

The platter is a metal disk coated with magnetic film. It spins constantly at 5400-7200 RPM (90-120 spins per second).

The head floats nanometers above the surface but never touches it (a head crash destroys the drive).

⚙️ The two physical delays you cannot escape

To read block X, the controller knows exactly which track (which ring) and which sector (which spot on that ring) holds X. But "knowing" is not "having". Two mechanical things must happen:

1. Seek time

The arm must physically swing from its current track to the target track. The arm has mass; it cannot teleport.

Typical: 4-10 ms for a random seek.

Already on the right track? Seek = 0.

2. Rotational latency

Once on the right track, you must wait for the right sector to rotate under the head. The platter spins at a fixed speed - you can't speed it up.

Average: 4 ms at 7200 RPM (half a rotation).

If the sector just passed by, you wait one full rotation (~8 ms).

🏃 Why continuous beats scattered (the lightbulb moment)

✅ Continuous: blocks 2, 3, 4

  1. Seek to track holding block #2: ~5 ms
  2. Wait for block #2 to rotate under head: ~4 ms
  3. Read block #2 (~0.1 ms)
  4. Block #3 is the very next sector on the same track. The platter is already spinning; #3 arrives under the head in ~0.05 ms. No new seek, no new rotation wait.
  5. Same for block #4.

Total: ~9 ms for the first, then ~0.1 ms each. ~9.2 ms total.

❌ Scattered: blocks 2, 7, 10, 5

  1. Seek + rotate + read block #2: ~9 ms
  2. Block #7 is on a different track. New seek (~5 ms) + new rotation wait (~4 ms) + read = ~9 ms
  3. Block #10 on another track: ~9 ms
  4. Block #5 on yet another track: ~9 ms

Total: ~36 ms. Roughly 4× slower for the same 4 blocks.

So knowing the address is necessary but not sufficient. Knowing where something is doesn't move the hardware faster. The hardware has to physically traverse to it, and that traversal is what dominates the time.

RAM analogy to make this concrete: RAM has no moving parts - electrical signals reach any cell in ~100 ns. It's like flicking a light switch. HDD is like a librarian on a unicycle riding around a circular library - even if she knows exactly which shelf to go to, she still has to pedal there.

💾 What about SSDs? (the modern reality)

Almost all modern laptops and servers use SSDs (Solid State Drives). Good news: SSDs have NO moving parts. No spinning platter, no swinging arm, no head. Data lives in NAND flash memory chips - basically billions of tiny transistors that hold charge.

So does the seek/scatter problem disappear?

✅ What SSDs fix

  • Random access is fast - ~50-150 microseconds (vs 5-10 ms on HDD). That's 50-200× faster.
  • No "seek time" - reaching any flash cell takes the same time as any other.
  • Scattered reads are much closer to sequential reads in performance.

⚠️ What SSDs do NOT fix

  • Still block/page based - the smallest read is one "page" (typically 4 KB or 16 KB). Want 1 byte? You still get 4096.
  • Writes are even more painful - SSDs erase in big "erase blocks" (~256 KB) before they can rewrite, leading to write amplification.
  • Each cell wears out after ~1000-100,000 writes; the SSD's controller does wear leveling by scattering writes deliberately.
  • Total throughput per query is still limited by how many block reads you do.

💾 What is "write amplification"? (the simple version)

Short definition: Write amplification means that when you ask an SSD to write a small amount of data, it ends up physically writing much more than that under the hood. The amount you wanted vs the amount actually written is the "amplification".

Why this happens (the 3 rules of NAND flash)

SSDs play by three weird rules that come from the physics of NAND flash cells:

  1. Reads happen at the page level (small - typically 4 KB or 16 KB).
  2. Writes also happen at the page level - BUT only to a page that's been freshly erased. You cannot overwrite a page that already has data.
  3. Erases happen at the erase block level, which contains many pages bundled together (typically 128-256 pages = 512 KB to 4 MB). The hardware physically cannot erase just one page.

So the painful combination is: you can't reuse a page until it's erased, and erasing takes out a huge group of pages at once.

A concrete example - updating 100 bytes

Say your SSD has 4 KB pages and erase blocks of 128 pages (= 512 KB). You want to update 100 bytes in the middle of a file.

Step-by-step: what the SSD actually does
  1. The 100 bytes you want to change live inside page #42. That page is inside erase block #3 (which has 127 other pages, all holding other data you don't want to touch).
  2. The controller cannot just rewrite page #42 in place - rule #2 forbids it.
  3. So the controller finds a different erase block that's already empty, say block #99.
  4. It reads page #42 into a buffer, modifies the 100 bytes, and writes the updated page to a free page in block #99. (1 page = 4 KB written so far)
  5. It updates an internal mapping table: "the logical page that used to live at block #3 / page #42 is now at block #99 / page #0".
  6. The original page #42 in block #3 is now marked as stale.
  7. Later, when block #3 has accumulated lots of stale pages, the garbage collector kicks in. It must reclaim block #3, but block #3 still has 127 valid pages mixed in with stale ones. So:
    • Copy all 127 valid pages to yet another free block (= 127 × 4 KB = 508 KB written)
    • Erase the entire 512 KB block #3

The math: you wrote 100 bytes from your program's perspective. The SSD physically wrote ~512 KB (1 page for the update + 508 KB during garbage collection). That's a write amplification factor of ~5000× in the worst case.

Real-world SSDs are smarter (write buffering, log-structured writes, over-provisioning, etc.) so the typical amplification is more like 1.5× to 5× - but it's never below 1, and on small-random-write workloads it gets ugly fast.

A simple analogy

Imagine a notebook where the only way to erase is to tear out an entire page (even if you only want to fix one word). To "edit" a word, you must:
  1. Copy every other sentence on that page to a fresh page
  2. Add your corrected word
  3. Tear out the old page
You changed 5 letters but rewrote a whole page. That's write amplification - your intended change is small, but the physical work the device must do is much bigger.

Why you should care

  • Wear: Every flash cell can only be erased ~1,000-100,000 times before it dies. A 5× amplification means your SSD wears out 5× faster than the raw byte count suggests.
  • Performance: When garbage collection runs heavily (common after lots of small random writes), the SSD slows down dramatically - sometimes 10× slower than its advertised speed.
  • DB design: This is why databases that batch many small writes into one big sequential write (LSM trees, write-ahead logs, etc.) perform much better on SSDs than databases that issue lots of tiny random updates.

💰 So does the DBMS choice change on SSD?

No - B-Trees still win, for the same reason. A B-Tree with fanout 500 needs only ~4 block reads for a billion-row table.

  • HDD: 4 reads × 10 ms = 40 ms
  • SSD: 4 reads × 100 µs = 0.4 ms

Compare to a sorted array with binary search (~30 reads):

  • HDD: 30 × 10 ms = 300 ms
  • SSD: 30 × 100 µs = 3 ms

Even on SSDs, B-Trees are ~7× faster than binary search on a sorted array. The principle "minimize block reads, pack data densely" holds whether the underlying storage is mechanical or solid-state. SSDs just shift the absolute numbers down without changing the ranking.

One SSD-specific twist: Some newer database engines (LSM trees in RocksDB, Cassandra, etc.) are designed specifically to favor sequential writes over random writes - precisely because of SSD write amplification. So "which structure is best" can depend on whether your workload is read-heavy (B-Tree wins) or write-heavy on SSD (LSM tree often wins).
🔦 Going deeper: why physically is RAM byte-addressable and disk isn't? And is scattered-page hopping really what causes the disk latency?

Yes, your intuition is exactly right. Let's nail it down with the underlying physics.

🧠 What makes RAM "byte addressable" - the physics

RAM is built from billions of tiny capacitors (DRAM) or flip-flops (SRAM). Each storage cell holds 1 bit. 8 cells together = 1 byte. Every byte has its own unique numeric address.

RAM internal organization (simplified)

When the CPU sends address 0x05 to RAM:

  1. The address is split into a row number and column number.
  2. The row decoder activates one row's word line. All cells in that row are now "open".
  3. The column decoder reads the bit from the right column's bit line.
  4. Out comes the byte at address 5 (the yellow cell).

This is pure combinational logic - electrical signals flipping gates. No mechanical motion. The whole operation takes ~100 nanoseconds.

⚡ Why every byte in RAM is equally fast

  • Electrical, not mechanical: Reaching cell #5 and cell #5,000,000 are both done by flipping transistors. Electrons move at near light speed. No physical part has to relocate.
  • Hardware address decoder is parallel: No matter what address you feed in, the decoder activates exactly one row and one column in the same number of gate delays. The decoder doesn't "look harder" for far-away addresses.
  • Wires are tiny: The chip is millimeters across. Signal propagation is essentially instantaneous at this scale.
  • Designed for byte access from day one: The smallest physically separable cell-group on a DRAM chip is one byte (8 bits sharing a column). The interface to the CPU exposes byte-level addresses directly.

(There's a tiny exception: DRAM has a "row buffer" that caches the last accessed row. Reading two bytes from the same row is a few nanoseconds faster than from different rows. But this is a 10-20 ns difference, not orders of magnitude - so for practical purposes, "all bytes equally fast" is true.)

🛡️ Why disk physically cannot be byte-addressable

Both HDDs and SSDs are forced into block-level granularity by their underlying physics:

HDD: spinning magnetic platters

  • Data is read by the head as the platter spins past. The head cannot "stop and look at one bit" - the platter never stops.
  • Each sector on the platter starts with a marker, contains the data, then ends with error-correction codes (ECC). The hardware needs the whole sector to verify integrity. Reading partial sectors makes no sense.
  • Sector size is baked into the firmware (typically 512 B or 4 KB).

SSD: NAND flash chips

  • NAND flash is physically organized into pages (4-16 KB). The smallest electrical read circuit operates on a whole page in parallel.
  • Like HDDs, ECC is computed per page. Reading less than a page gives unreliable bits.
  • The bus between SSD controller and flash chip is wide and designed to transfer a full page per cycle.

So the constraint is not laziness or convention - it's the physical structure of the storage medium. Even if the CPU sent a "give me 1 byte" command, the underlying chip cannot fulfill it without reading the full block.

✅ Your understanding confirmed: yes, scattered pages cause the latency

You wrote: "does latency in disk even if data is sorted comes because pages containing data of single table may not present in continuous memory blocks and hopping from one page to another is time taking operation when they are scattered over the memory"

That is exactly right. Let's restate it precisely:

  1. "Sorted" only means logical order - the rows ARE in ascending key order if you walk them in file-offset order.
  2. But the file's bytes are stored in disk pages, and those pages are physically scattered across the drive (because of file growth, deletes, the OS allocating wherever there's free space).
  3. Each time binary search jumps to a different file offset, it lands in a different physical page.
  4. Each new page = a new physical read operation:
    • HDD: seek (~5 ms) + rotational wait (~4 ms) + transfer = ~10 ms per hop
    • SSD: page read (~100 µs) per hop - still not free, just much cheaper
  5. 30 hops × 10 ms = 300 ms on HDD. That's the latency you're feeling.
The cleanest way to remember it:

RAM: any byte costs the same, because access is purely electrical signal routing.

Disk: the smallest unit you can read is one block (forced by physics), AND each block read costs a separate physical operation (seek+rotate on HDD, page read on SSD). So fewer block reads = faster, and that's why B-Trees - which let you find anything in 3-4 block reads - dominate disk-based storage.

🧠 Connecting the dots back to your faculty's point

  1. Byte-addressable RAM: any byte is equally fast. arr[i] doesn't care if memory is "scattered" because there is no seek cost.
  2. Block-addressable disk: must read whole blocks AND blocks can be physically anywhere AND seeking between them is slow.
  3. Together this means: a "sorted array on disk" is a lie - it's logically sorted but physically scattered, and binary search becomes 30 expensive seeks.
  4. So we need a structure where: (a) one read pulls a useful chunk, and (b) we make as few reads as possible. That's a B-Tree.
One more wrinkle: Modern SSDs don't have a mechanical head, so the seek penalty is way smaller (~100 microseconds vs 10 ms). But blocks (called "pages" on SSDs, usually 4 KB) are still the minimum unit, so the "read 4096 bytes to get 1 byte" problem stays. B-Trees still win on SSDs because they minimize the total number of block reads.

Naive approach #3: Hash table on disk

O(1) on average, but no range queries (WHERE age BETWEEN 20 AND 30), and hash collisions force chasing pointers across many blocks.

What we need:
  • Few disk reads (each read is expensive)
  • Each disk read should be information-dense - pack lots of useful data per block
  • Support range queries and ordering
  • Stay balanced under inserts and deletes

B-Tree (and B+ Tree) Explained

A B-Tree is a self-balancing tree where each node can hold many keys and many children (not just 2 like a binary tree). The "B" was coined by its inventors (Bayer & McCreight, 1972) and is often said to mean "balanced", "broad", or "Bayer".

Is it used by relational DBMS?

Yes - almost universally. But technically, most relational databases use a variant called B+ Tree:

  • MySQL (InnoDB) - B+ Tree for primary & secondary indexes
  • PostgreSQL - B+ Tree (default index type)
  • Oracle, SQL Server, SQLite - all use B+ Trees

B-Tree vs B+ Tree (the small but crucial difference)

B-Tree

  • Data stored in all nodes (internal + leaf)
  • Finding a key can stop early at an internal node

B+ Tree (used by DBMS)

  • Actual row data only in leaf nodes
  • Internal nodes hold only keys & pointers - so they're tiny and fit more keys per block
  • Leaves are linked together - perfect for range queries like BETWEEN

Structure rules (order m)

  • Every node has up to m children and up to m-1 keys
  • Keys inside a node are sorted
  • All leaves at the same depth (balanced)
  • m is chosen so one node = exactly one disk block (typically m = 100 to 500)

How data is stored - a concrete example

Say InnoDB has a 16 KB page and rows are ~160 bytes. A leaf page holds ~100 rows. An internal page holds maybe 500 (key + pointer) entries because they're smaller.

Internal Node (one disk block, 16 KB):

Each ptr points to a child node (also one disk block).

Leaf Node (one disk block, 16 KB):
[ row(id=101, name=..., email=...) | row(id=102, ...) | ... ~100 rows ]
[ next_leaf_ptr -> ]   <-- linked list for range scans

Why B-Trees Win on Block-Addressable Disk

1. One node = one block

Reading one B-Tree node = exactly one disk I/O. The tree is designed around the disk's read unit, not against it.

2. High fanout = shallow tree = few disk reads

If each node has 500 children, then in just 4 levels you can index 500⁴ = 62.5 billion rows.

Height 1: up to 500 rows
Height 2: up to 250,000 rows
Height 3: up to 125,000,000 rows
Height 4: up to 62,500,000,000 rows  <-- typical max for huge tables

So searching a billion-row table needs only ~4 disk reads. At 10 ms each, that's 40 ms total. Compare to 6 days for linear scan.

3. Sequential leaf links = cheap range queries

In a B+ Tree, leaves form a linked list. So WHERE id BETWEEN 100 AND 1000 finds the start leaf in 4 reads, then walks the linked list - each step is one block read covering ~100 rows.

4. Top of tree stays in RAM (buffer pool)

The root and the next 1-2 levels are tiny (a few MB total) and get cached in RAM by the DBMS's buffer pool. So in practice the root and second-level reads cost 0 disk I/O. Only the leaf read actually hits disk.

Effective disk reads per lookup: often just 1.

5. Inserts stay balanced

When a node overflows, it splits and pushes a key up. This keeps the tree balanced without expensive global reorganization - very different from a sorted array on disk.

Summary of why B-Trees fit block-addressable storage so well:
  • Each node is sized to exactly one block - no wasted I/O
  • Fanout is huge, so trees stay shallow - O(log500 n) instead of O(log2 n)
  • Internal nodes are cached aggressively in RAM
  • Leaf links enable efficient range scans (B+ Tree)
  • Self-balancing keeps performance predictable as data grows

So when your faculty said...

..."databases use special data structures because disk is block addressable" - they meant: the algorithm and the storage hardware are co-designed. B-Trees aren't faster because they're a clever idea in isolation. They're faster because they take the disk's worst limitation (must read whole blocks) and turn it into a strength (each block read tells you about ~500 keys at once).

High-Level Design of a Relational DBMS

High level architecture of relational DBMS in three points like Postgres or MySQL.:
  1. ✅ "Main file where data is stored in optimised form" - correct. This is the data file (heap files + B-Tree index files on disk).
  2. ✅ "Buffer pool in RAM holds latest reads/writes" - correct. Writes don't go straight to disk because one logical change touches many physical structures (data page, multiple indexes, metadata) and a crash mid-way would corrupt the database.
  3. ✅ "WAL exists on disk, sequential writes = fast" - correct. Small correction: the data isn't pushed from the WAL to the main file. The buffer pool holds the modified pages; the WAL just records the intent. Dirty pages from the buffer pool are written to the main file at checkpoints, and only then can the corresponding WAL records be discarded.

🏗️ The component diagram

Inside a Relational DBMS

✏️ The write path - what happens on UPDATE users SET name='X' WHERE id=42

  1. User issues UPDATE.
  2. Database locates the target row (often via an index).
  3. Required locks / MVCC metadata are acquired.
  4. Relevant page(s) are loaded into the buffer pool if not already cached.
  5. Constraints are validated.
  6. Table and possibly index pages are modified in memory (buffer pool).
  7. Dirty pages are created.
  8. WAL records describing the changes are generated and appended to the WAL buffer.
  9. Commit record is written to the WAL.
  10. WAL is flushed to durable storage (fsync).
  11. Transaction is acknowledged as committed to the client.
  12. Dirty table/index pages are flushed to disk later by checkpoints / background writers.

📖 The read path - what happens on SELECT * FROM users WHERE id=42

  1. Parse + plan + execute, same first 3 steps as above.
  2. Executor needs the leaf page that contains id=42. It asks the buffer pool first.
  3. Cache hit: page is already in RAM → return in microseconds. (For the root and upper levels of the B-Tree, this is almost always the case.)
  4. Cache miss: buffer pool issues a block read from the data file. If the buffer pool is full, it evicts some other page (LRU-ish policy). If the evicted page was dirty, it must be written back to disk first - but only after its WAL records are flushed.
  5. Return row to client.

📁 Checkpoints - when does WAL actually get freed?

WAL files grow forever otherwise. A checkpoint is a periodic event (every few seconds to minutes) where:

  1. All currently-dirty pages in the buffer pool are flushed to the main data file.
  2. Once flushed, a "checkpoint marker" is written to the WAL.
  3. WAL records older than the most recent checkpoint can now be discarded (truncated).

So the WAL doesn't fill up because "buffer pool is full" - rather, the checkpoint process keeps both in balance. Your "WAL is full → push to main file" mental model is in the right direction but the trigger is checkpoints (time/log-size based), not the buffer being full.

🔨 Crash recovery - the WAL's whole reason to exist

Suppose the server crashes. RAM is gone, so the buffer pool's dirty pages vanish. On restart:

  1. Analysis phase: read the WAL to figure out which transactions were committed at the time of crash.
  2. Redo phase: walk through the WAL from the last checkpoint forward. For every committed change whose effect might not have made it to the data file (because the dirty page hadn't been flushed yet), re-apply the change to the data file.
  3. Undo phase: for transactions that started but never committed, roll back their effects using the WAL's "before image" records.

End result: the database is restored to a consistent state matching exactly what was committed before the crash. This is the D in ACID (Durability).

🏃 Why sequential WAL writes are so much faster than random data-file writes

Recall everything we learned:

  • A single user transaction may touch many pages (data + multiple indexes), scattered across the data file.
  • If we wrote every change directly to the data file, we'd do many random block writes - ~10 ms each on HDD.
  • Instead, we write all the change records in order, append-only to the WAL. That's one continuous stream → the disk head doesn't move (or for SSD, no random-write amplification).
  • The slow "scatter the dirty pages to their proper homes" work is batched at checkpoint time and done in the background.

So WAL is the trick that lets transactions commit fast while still being durable. The slow disk work happens later, asynchronously, without making the client wait.

🧮 Putting it all together: ACID via these components

Ⓜ Atomicity

All-or-nothing. Guaranteed by WAL's undo records + transaction IDs. If a transaction never commits, recovery rolls back any partial changes.

ⓒ Consistency

Constraints, foreign keys, triggers - enforced by the executor before WAL records are written.

Ⓜ Isolation

Concurrent transactions don't see each other's half-done work. Lock manager (2-Phase Locking) or MVCC (versioned rows) achieves this.

Ⓦ Durability

Once COMMIT returns, the change survives crashes. WAL fsync before COMMIT + recovery on restart give this guarantee.

📝 Cleaned-up version of your mental model

  1. Data files on disk hold the actual rows in B-Tree organized form (one heap file per table, one B-Tree file per index).
  2. Buffer pool in RAM caches pages. Both reads and writes go through it. Modified pages stay "dirty" in RAM and are written back to disk lazily.
  3. Write-Ahead Log (WAL) on disk is a sequential, append-only file. Every change is logged here before it's allowed to be flushed to data files. WAL writes are cheap because they're sequential.
  4. Checkpoints periodically flush all dirty pages to the data files and then truncate the WAL. This is when the "expensive random write" work gets batched.
  5. On crash: replay the WAL from the last checkpoint to redo committed work and undo uncommitted work. The database comes back consistent.

💡 Why Updating a Page in RAM is Faster Than Updating it on Disk

Take the same 8 KB B+ tree page. It exists on disk. It's also loaded into the buffer pool in RAM. You want to change one value in it. Here's what each path costs:

Updating the page in RAM
  1. CPU sends a write instruction to the memory address.
  2. Electrical signal activates the row + column decoder inside the RAM chip.
  3. The target cell is overwritten.

Total time: ~100 nanoseconds. Fully electrical, no physical movement.

Updating the page on disk (HDD)
  1. OS issues a seek: the arm physically swings to the right track (~4–8 ms).
  2. Wait for the platter to rotate the right sector under the head (~4 ms at 7200 RPM).
  3. Read the old block into RAM (disk can only write whole blocks).
  4. Modify the bytes in RAM.
  5. Write the full block back to disk (~1 ms sequential write).

Total time: ~10 milliseconds. Dominated by mechanical movement.

That's a 100,000× difference — not because the CPU is doing different work, but because the medium forces different physical constraints:

  • RAM is electrical. Any address is reached by flipping transistors. There is no "distance" between addresses — everything takes the same ~100 ns.
  • Disk is mechanical (HDD) or has erase constraints (SSD). An HDD must physically move metal to the right spot. An SSD can't overwrite in-place — it must erase a whole flash block (~512 KB) before writing, even if you changed 8 KB. Both are orders of magnitude slower than RAM.
  • Disk is block-addressable. Even if the mechanical problem didn't exist, disk hardware can only read/write in fixed block sizes. Changing one byte still requires reading the whole block, changing it in RAM, and writing the whole block back. RAM lets you write exactly the bytes you want.
The one-sentence answer:

Updating a page in RAM is faster because RAM is a pure electrical circuit — any write lands in ~100 ns with no physical movement and no block-size constraints — while a disk write triggers a mechanical seek (HDD) or an erase-before-write cycle (SSD) that takes ~1,000–100,000× longer.

The buffer pool exists precisely to exploit this gap: keep the page in RAM, do all your modifications there at electrical speed, and only pay the disk cost once — in bulk, at a checkpoint, when thousands of changes can be written in a single sorted pass.
